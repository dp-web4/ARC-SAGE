# sb26 — Sequence-matching puzzle with colored tokens placed into frames

## Toolset
Available actions: [ACTION5, ACTION6, ACTION7]. Click-based (ACTION6) with a "play/evaluate" action (ACTION5) and undo (ACTION7). This is a placement-then-verification game: arrange colored tokens in the bottom row, then press play to see if the sequence matches target slots in frames above.

- `vgszefyyyp` (tag: `lngftsryyw`, `sys_click`) — Portal/warp token (6x6, ring-shaped with inner color). Interactive. When the evaluator reaches this tile, it warps to a matching-color frame.

## Interaction Model
- **CLICK (ACTION6):** Click a token in the bottom row to select it (highlighted). Click another token to swap them. Click an empty spot inside a frame to move the selected token there (and vice versa). Two tokens in the bottom row swap positions; token + empty-slot moves the token. Each swap/move costs 1 energy. Clicking the same token deselects.
- **PLAY (ACTION5):** Starts the automated evaluation sequence. The system reads frame contents left-to-right, matching tokens against the target color sequence at the top. Costs 1 energy. Triggers an animated multi-step process with cursor movement, color filling, and success/failure animations.
- **UNDO (ACTION7):** Reverts the last placement/swap. Maintains a state history stack (`uvawsoycr`).

## Win Condition
In `dbfxrigdqx()`: The game wins when `pmygakdvy == len(wcfyiodrx) - 1` and the current target slot is already filled (`wrudcanmwy` is true). In plain English: **all target color slots have been successfully matched** by the token sequence in the frames. The evaluator reads tokens from frames in order, and each token's color must match the next target slot. Portal tokens (vgszefyyyp) redirect evaluation to matching-color frames. `next_level()` is called on success.

## Evaluator Walk (engine `dbfxrigdqx`)
Evaluation is a stack-based recursion over frames:

- `buvfjfmpp` = call stack of (frame, slot_index).
- `pmygakdvy` = current target index (0-based).
- At each tick, read the item at `rzbeqaiky[frame][slot_index]`:
  - **Regular token** (`lngftsryyw`): color must equal `wcfyiodrx[pmygakdvy].pixels[0,0]`. If yes, fill slot (xjxrqgaqw animation) and advance. If no, `sibihgzarf()` fails the run.
  - **Portal** (`vgszefyyyp`): look up frame whose border color matches portal inner color, push (that_frame, 0). Portal itself consumes no target.
  - **Pre-placed token in a frame** counts just like a placed regular token during evaluation.
- After advancing within a frame, if slot_index == frame_width, pop. If stack becomes empty with targets remaining, fail.
- **Win test** (line 926): fires exactly when `pmygakdvy == len-1` AND the current target slot is already filled. So the LAST target must match via a regular-token read.
- **Portal re-entry guard** (line 976): fails if pushing an already-at-slot-0 frame when the immediately-prior stack entry is also at slot 0. Narrow — does not prohibit all re-entry.

## L4+ Mechanic Clarification
The original solver assumed 1:1 mapping from palette row to first-frame slots, and failed whenever multi-frame routing was required. **Total slot count = sum of empty slots across ALL frames reachable via portals, not just the first frame.** Key behaviors that drove L4–L8 failures:

- **L4**: 7 targets, 5 top-frame slots + 2 bottom-frame slots = 7 (bottom frame has 1 pre-placed identity token so only 2 are empty). You must place a portal in the top frame (its inner color matches the bottom frame's border) to reach the bottom slots. "border_color_identity" in `sb26_solutions.json` = the bottom frame's pre-placed token matches its border color.
- **L5**: Two portals in palette, three non-first frames with pre-placed identity tokens. Pattern: "multi_parent_slots" — border color occupies multiple parent slots via portals.
- **L6**: Four frames, all initially empty; two or three portals in palette. "swap_grid_container" was a misnomer — same pick-and-place interaction, just more complex routing.
- **L7**: Nested / palindrome-shaped recursion: top frame contains a portal that lands in a frame which itself contains a portal back/onward.
- **L8**: Two frames, each ends with a portal to the other — the evaluator bounces between them producing a doubled sequence (targets appear twice). Re-entry is allowed because pushed-frame slot indices differ from 0 after first entry.

The correct solver approach: **simulate the engine evaluator** and DFS over palette-to-slot assignments, picking for each empty-slot position either a color-matching regular token or a portal whose sub-walk yields the needed suffix. This solves all 8 levels in 124 actions (baseline 153).

## Level Progression
**8 levels.** Grid size is always 64x64.
- Level 1: 4 tokens, 1 frame (4 slots), 4 target colors. Simple direct matching.
- Level 2: 7 tokens, 2 frames, 7 targets. Introduces portal/warp tokens connecting frames.
- Level 3: 8 tokens, 2 frames, portal tokens.
- Level 4: 8 tokens, 2 frames, portal tokens.
- Level 5-6: 10 tokens, multiple frames with more complex portal chains.
- Level 7: 8 tokens, complex warp paths.
- Level 8: 12 tokens, 2 rows of 6 targets each, multiple frames.

Key level data: sprite positions, token colors, frame sizes (determined by sprite name suffix digit, e.g., `qdmvvkvhaz4` = 4-slot frame).

## No-Source Discovery Strategy
1. **First click:** Click a colored square in the bottom row. Observe a highlight appears.
2. **Observe:** Click another colored square -- they swap positions.
3. **Hypothesis:** Bottom row tokens can be rearranged by selecting and swapping.
4. **Test ACTION5:** Press play. Watch the animated evaluator move through frames, filling target slots at the top. If a color matches, it fills in; if wrong, red flash and reset.
5. **Key insight:** Ring-shaped tokens are portals that jump evaluation to another frame. Arrange tokens so the full read-through matches the target sequence at top.

## Key Patterns
- Energy bar at row 53 -- each swap and each play costs 1 energy unit. Starting energy: 64. Lose when energy hits 0.
- Tokens are spaced 6 pixels apart in frames (`kojduumcap = 6`).
- Frame content offset is 2px from frame edge.
- Evaluation is animated: 6 frames for cursor movement, flash animations for success/failure.
- Portal tokens have a ring shape (inner color visible at pixel [1,1]). The evaluator jumps to the frame whose border color matches the portal's inner color.
- The evaluator can chain through multiple frames via portals, building a path that must produce the exact target color sequence.
- Wrong color triggers 18-frame red flash animation, then resets evaluation.
- Color palette: 0 (black/bg), 8 (azure), 9 (maroon), 11 (teal), 12 (pink), 14 (grey), 15 (white). Background=4 (yellow), padding=3 (green).
