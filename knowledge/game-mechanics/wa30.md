# wa30 -- Grid-based pathfinding puzzle with paired agents and target zones

## Toolset
Available actions: [1, 2, 3, 4, 5] (UP, DOWN, LEFT, RIGHT, SELECT/INTERACT).
This is a directional movement game on a discrete grid with a step budget. The 5th action is context-dependent interaction, not movement.

## Objects
All sprites occupy a 64x64 grid at step size `celomdfhbh = 4` (grid cells are 4px apart).

## Interaction Model
- **ACTION1 (UP)**: Move player 4px up (dy = -4). Step counter decrements.
- **ACTION2 (DOWN)**: Move player 4px down (dy = +4). Step counter decrements.
- **ACTION3 (LEFT)**: Move player 4px left (dx = -4). Step counter decrements.
- **ACTION4 (RIGHT)**: Move player 4px right (dx = +4). Step counter decrements.
- **ACTION5 (SELECT)**: Toggle attachment. If player is attached to a follower, detach. If player is adjacent (4px manhattan distance) to an unattached target slot, attach player to it. If adjacent to secondary follower, remove it.

After each player action, `dhrikuybfo()` runs the AI step:
1. `ynmgxjqkgh()` -- Follower agents (`kdweefinfi`) BFS pathfind toward nearest unoccupied target slot. When adjacent, they attach. When attached, they try to reach docking zone A (`fsjjayjoeg`).
2. `aoeyzovteg()` -- Secondary followers (`ysysltqlke`) do the same toward docking zone B (`zqxwgacnue`).
3. `zzppkjnqgk()` -- Updates visual feedback (border colors) on target slots to indicate attachment status.

Player faces movement direction via rotation (0=up, 90=right, 180=down, 270=left). The facing direction determines which adjacent target the SELECT action interacts with (`vwiozbtqgi` checks the cell in the facing direction).

Paired movement: when player is attached (`nsevyuople` dict), both player and attached sprite move together maintaining their relative offset.

## Win Condition
Method `ymzfopzgbq()`: returns True when ALL sprites tagged `geezpjgiyd` (target slots) are positioned within `fsjjayjoeg` (docking zone A) coordinates AND are NOT currently attached to any follower.

In `step()`:
1. If `ymzfopzgbq()` is true -> `next_level()`
2. Else if step counter reaches 0 -> `lose()`

## Level Progression
9 levels. Grid size is always 64x64. Each level has:
- A `StepCounter` value (level 1: 200 steps, most others: 70 steps)
- Varying numbers of target slots, followers, walls, and docking zones
- Increasing complexity: more targets, more walls, narrower paths

## No-Source Discovery Strategy
1. Move in each direction (4 actions) to confirm it is a directional movement game with step budget bar at bottom.
2. Walk toward a bordered square (target slot) and press SELECT when adjacent to see attachment behavior.
3. Observe AI agents moving autonomously after each player move.
4. Try detaching (SELECT again) and observe the slot sprite moving on its own.
5. Move target slots into the large bordered rectangles (docking zones).
6. Watch for color changes on borders to understand correct/incorrect placement.
7-10. Experiment with multiple targets and second agent type.

## Key Patterns
- The game is a Sokoban-like puzzle where you guide objects to zones, but objects also have autonomous pathfinding agents that move them.
- Player attachment toggles with SELECT based on facing direction.
- Step budget is the primary constraint; efficient pathfinding is critical.
- Follower agents take 1 BFS step per player move, so player moves indirectly control AI progress.
- Two types of followers target two types of zones; mismatched followers block slots.
- Win requires ALL target slots in docking zone A with NO follower attachments remaining.

## Solving Notes (2026-04-11, mcnugget)

### Key Mechanics Verified from Source
- **SELECT (ACTION5)**: When adjacent to goal AND facing it (`vwiozbtqgi` checks player rotation), attaches goal to player. Player carries goal with fixed offset (direction faced × 4px). SELECT again drops.
- **Win = `ymzfopzgbq()`**: ALL goals at dock zone pixels (`wyzquhjerd` from `fsjjayjoeg` sprites) AND not claimed (`zmqreragji`).
- **Carry offset**: Goal stays at (player.x + dx, player.y + dy) where (dx,dy) = direction player faced when picking up × STEP.
- **Blocking**: Goals in `pkbufziase` block all movement. Delivered goals block paths to remaining goals.
- **No walls in L1**: Only border squares block. L1 is pure pick-and-deliver.
- **`perform_action` compatible**: No collision-state issues like s5i5.

### L1 Layout
- Player: (32,48). Goals: (16,28), (32,36), (44,24). Dock zone: (28,28) 12×4 = x=28-39, y=28-31.
- 3 dock positions needed at 4px grid: e.g., (28,28), (32,28), (36,28).
- Delivery ORDER matters: delivered goals block paths to remaining goals.

### Approach for Solver
Need a planner that: 1) assigns goals to dock positions, 2) determines delivery order to avoid blocking, 3) pathfinds around delivered goals. BFS too expensive with deepcopy.
