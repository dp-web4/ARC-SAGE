# lf52 -- Grid-based puzzle with tile movement, merging, and custom rendering engine

## Toolset
Available actions: [ACTION1 (UP), ACTION2 (DOWN), ACTION3 (LEFT), ACTION4 (RIGHT), ACTION6 (CLICK), ACTION7 (UNDO)]
This game uses a custom internal scene graph (`rjfmjxejeiq`) rather than standard ARC sprite positioning. It has its own animation framework, camera scrolling, and tile-based grid. UNDO is available when `STORES_UNDO = True`.

## Objects
The game uses a scene graph with named nodes rather than sprite tags. Key object types:
| Name pattern | Role |
|---|---|
| `fozwvlovdui` | Primary game tiles -- clickable, movable puzzle pieces |
| `fozwvlovdui_red` | Red variant tiles |
| `fozwvlovdui_blue` | Blue variant tiles (static in levels 8+) |
| `lgbyiaitpdi` | Direction indicator arrows (4 cardinal directions) |
| `hupkpseyuim` / `hupkpseyuim2` | Grid cell markers -- define the playable grid |
| `kraubslpehi` | Walls/boundaries within the grid |
| `cwyrzsciwms` | Special clickable trigger (activates `zvcnglshzcx` flag) |
| `bhdfjlqapap_0/1` | Tutorial/hint animation sprites |
| `csrvckunbev` | Player cursor/selector |

Color palette: BACKGROUND_COLOR = 0 (LIGHT_BLUE mapped to 0), custom rendering via scene graph.

Camera: `Camera(background=0)` with custom `wnrtmoelug` interface for frame rendering. The game renders frames via `equnaohchtj.render()` and passes them to the camera interface.

## Interaction Model
- **ACTION1 (UP)**: Increments step counter. Calls `tmhxwcojkh(0, -1)` -- moves `hupkpseyuim2` tiles upward. Tiles are sorted by grid_y (descending for up). If a tile would move into a `kraubslpehi` wall, it stops. If it meets another `hupkpseyuim2` tile, it passes through. Moving a `fozwvlovdui` tile into a wall may trigger camera scrolling (`hncnfaqaddg.cdpcbbnfdp` adjustment) with animated panning, varying by level.

- **ACTION2 (DOWN)**: Same as UP but `tmhxwcojkh(0, 1)`.
- **ACTION3 (LEFT)**: `tmhxwcojkh(-1, 0)`.
- **ACTION4 (RIGHT)**: `tmhxwcojkh(1, 0)`.

- **ACTION5** (mapped but calls `kuexigxyxw` which just calls `pchvqimdvj()` then passes -- likely a "confirm" or no-op in current implementation).

- **ACTION6 (CLICK)**: Converts click coordinates to scene graph position via `rozmjeilxc(x, y)`. Behavior depends on what's clicked:
  - Clicking a `fozwvlovdui` tile: selects it and shows direction arrows (`lgbyiaitpdi`) for valid movement directions.
  - Clicking a `lgbyiaitpdi` arrow: moves the selected tile in that direction. The tile slides to a new grid position, and if a same-colored `fozwvlovdui` tile is in the path, they merge (the intermediate tile is removed with a shrink animation).
  - Clicking `cwyrzsciwms`: activates a special undo/reset button that appears in the bottom-left corner.
  - Clicking the bottom-left area when `zvcnglshzcx` is active: triggers a full level reset animation.
  - Clicking empty space: clears the selection/arrows.

- **ACTION7 (UNDO)**: Calls `aphzlzlgni()` which restores the previous state from the undo stack. Increments step counter.

- **Animation system**: Fully custom. The game builds frame lists (`list[np.ndarray]`) and passes them to the camera interface. The `step()` method checks `xcjqviblhlt` flag -- on first call it processes the action and generates frames, on subsequent calls it drains the frame queue until animation completes.

## Peg-solitaire ground truth (decoded April 2026, CBP)

The surface appearance of "tile merging" is actually **peg-solitaire with a push-transport layer**.

- Pieces: `fozwvlovdui` (neutral/N), `fozwvlovdui_red` (red/R), `fozwvlovdui_blue` (blue/B).
- Static pegs: `dgxfozncuiz`.
- Pushable blocks: `hupkpseyuim2` — slide through adjacent `kraubslpehi*` wall cells when ACTION1-4 is pressed. Multiple blocks move in one push (sorted in push direction).
- A piece riding on a block (cells where block is one of the objects) is carried by the push.
- `dgxfozncuiz` on a block-cell (`P`/`D` glyphs) also rides the push.
- `hupkpseyuim` (walkable tiles) do NOT move with blocks in practice — blocks in this game always spawn on wall cells, never on walkable-only cells.

**Jump rule** (`qikmikecdf`): selecting a piece and clicking an arrow fires a 2-cell jump. Middle cell must contain any `fozwvlovdui*` OR `dgxfozncuiz` (`pymqmlkgzs`). Landing cell must be `len==1 and name contains hupkpseyuim` OR `len==2 and one is hupkpseyuim2` (`posalhhmjq`). A piece on a landing cell makes it invalid.

**Removal rule** (`cfilhtifcb` line 5393-5396): after a jump, if the middle cell contains a non-blue `fozwvlovdui*` piece whose `.name` matches the jumper's `.name`, that middle piece is removed. Blue middles are never removed; different-type matches (N vs R) are never removed.

**Blue is movable, not static.** This was the April 2026 model bug: prior solver excluded blue from `get_valid_jumps`. Blue pieces CAN be clicked, CAN be moved (via any direction where `qikmikecdf` holds), and — as either jumper or middle — participate in all landing/jumpable-middle checks. Blue moves just don't change the win counter. Once this bug was fixed, L8 and L9 collapsed from "0 reducing jumps anywhere in the reachable push+jump space" to solvable in seconds, because blue pieces often sit between two neutrals and are the only jumpable middle in the path.

**Win counter** (`ddaguepwkt`): `len(ndtvadsrqf("fozwvlovdui"))` — prefix-match includes N+R+B. For levels ≥ 8, subtract `self.lzoqlpcwzpu` (initial blue count, which doesn't change since blues can't be removed in normal play with no matching jumps). Win triggers when counter reaches 1 (or 2 for L6/L7) after a reducing jump at line 5572.

**L6 special win** (line 5472-5474): clicking the neutral piece at grid (16,2) while red is parked at (6,6) triggers `pchvqimdvj()` — a position-based win that doesn't require a jump. L7 has no analogous position-based win trigger; its win is purely counter-based.

**L7 unresolved (April 2026, re-investigated 2026-04-12)**:

Grid layout (23x9): two N pieces at (0,1) and (22,6), one R at (6,1). Target count = 2, requires ONE reducing jump. Since there is only 1 R, the only possible reduction is N-over-N (needs jumper-N + middle-N + valid landing).

New findings (2026-04-12 session):
1. **Model matches engine exactly on pushes.** Side-by-side probe on all 4 push directions from fresh L7 state: pieces, blocks, walls, walkable all identical between simulator and `equnaohchtj` instance. The simulator `apply_push` is faithful.
2. **Left-N@(0,1) CAN move** — block at (5,5) pushed via L,L,U,U,L,L,L reaches (0,3), making (0,2) peg a valid jumpable-middle and (0,3) a valid landing. Left-N can then jump (0,1)->(0,3). From there, left-N reaches 21 cells in cols 0-6. Red reaches 20 of the same cells. So left-N and R can swap positions freely on the left side.
3. **Right-N@(22,6) is permanently immobile.** Isolated BFS starting from right-N-only state (left-N and R removed) explored 500K states — right-N never leaves (22,6). `get_valid_jumps()` returns `[]` from its initial state. Its adjacent cells: (22,5) walkable-only (not jumpable), (21,6)/(22,7)/(23,6) empty/out-of-bounds (no object at all, can't be approached or made jumpable since they have no walls for blocks to occupy).
4. **Consequence**: N-over-N is impossible because left-N reaches max column 6 and right-N stays at column 22. They can never be adjacent. No reducing jump exists in the reachable state space.
5. **Win triggers re-verified**: Only `tdcblgbfxw` calls `self.win()`, and is reached only from `cfilhtifcb`'s counter check at line 5572 (`ddaguepwkt == 2` for L6/L7). `pchvqimdvj` does NOT call `win()` — it only plays animation and calls `cncmupctrp` which sets the zvcnglshzcx reset-button flag. ACTION5 also only calls `pchvqimdvj`, not a win. So the counter check is the only path to victory, and it requires a reducing jump.
6. **Interpretation**: Under the fully-decoded model, L7 appears structurally unsolvable. This is a strong contradiction since the game is distributed as playable. Remaining hypotheses:
   - (a) A push or jump mechanic exists that changes walkable/wall membership dynamically (e.g. a block entering a walkable+block cell creates a bridge), but the probe found no such mutation.
   - (b) ACTION6 clicks on something non-standard trigger hidden paths (e.g. clicking the scroll trigger regions when zvcnglshzcx is not yet set).
   - (c) There is yet another position-based win I haven't traced — possibly related to the scroll-animation triggers at (8,8)/(5,5) and col-16/(-39,5) that chain into something downstream.
   - (d) A peg-removal mechanic hidden elsewhere — but `dgxfozncuiz` never appears in `faretcdgmc` calls in the source.

Productive failure boundary: the previous hypothesis ("maybe push connectivity exists but exhaustive search was incomplete") is REFUTED. Left-N's reachable set is fully mapped (22 cells, cols 0-6 only). The new boundary is "what makes right-N movable, since push+jump alone cannot?"

## Win Condition
The `equnaohchtj` scene graph sets `iajuzrgttrv = True` via its `win()` method. This is checked after animation completes:
```python
if self.ikhhdzfmarl.iajuzrgttrv:
    self.next_level()
```
The win condition is triggered internally by the scene graph when all `fozwvlovdui` tiles have been merged/cleared from the grid (or a target configuration is reached). The exact trigger is inside `pchvqimdvj()` and level-specific position checks.

**Lose conditions**:
- `evxflhofing` flag set (explicit lose state)
- Level 1: step counter (`asqvqzpfdi`) >= 64
- Levels 2-5: step counter >= 64 * 5 = 320
- Levels 6+: step counter >= 64 * 10 = 640

## Level Progression
10 levels. Grid cell size is 6 pixels. Changes:
- Level 1: Small grid (10x5 offset), tutorial hints appear, simple merge puzzle
- Level 2: 6x8 grid offset
- Level 3: 5x5 grid offset, camera scrolling on tile push
- Level 4-5: Directional push variations
- Level 6-7: More complex scrolling mechanics
- Level 8+: Blue tiles introduced (static, counted separately)
- Level 9-10: Full complexity with combined mechanics

Step limits scale with level: L1 has 64-step limit, later levels allow 320 or 640 steps.

## No-Source Discovery Strategy
1. CLICK on colored tiles on the grid -- observe selection highlighting (1-2 actions)
2. CLICK on direction arrows that appear -- observe tile sliding (1-2 actions)
3. Slide a tile into a same-colored tile -- observe merging animation (1-2 actions)
4. Try directional keys (UP/DOWN/LEFT/RIGHT) -- observe bulk tile movement (1-2 actions)
5. Try UNDO (ACTION7) to revert a move
6. Notice camera may scroll when tiles are pushed near edges
7. Count remaining tiles to understand the objective (merge all same-colored pairs)

## Key Patterns
- Core mechanic is a tile-merging puzzle: slide tiles to merge same-colored pairs.
- CLICK-based selection + directional movement. Click tile, then click arrow to move it.
- Directional keys move ALL `hupkpseyuim2` tiles simultaneously in that direction.
- Camera scrolls dynamically based on level-specific rules when tiles hit boundaries.
- Merging removes one tile via animation. Goal is likely to clear/merge all tiles.
- Undo is available -- use it when moves lead to dead ends.
- Step limit varies by level. Early levels are tight; later levels are more generous.
- The bottom-left corner has a special reset trigger (click `cwyrzsciwms` then click bottom-left).

## L7 Investigation (2026-04-12) — Unresolved

**Status**: L7 remains unsolved. Two separate investigations converged on the same boundary without finding a resolving mechanic.

**Engine facts confirmed by source read + live probe**:
- L7 grid: 3 pieces: `fozwvlovdui` @(0,1) "left-N", `fozwvlovdui_red` @(6,1) "red", `fozwvlovdui` @(22,6) "right-N".
- Win condition (line 5572): `ddaguepwkt == 1 or (whtqurkphir in [6,7] and ddaguepwkt == 2)`.
- `ddaguepwkt` (line 5375) is computed via `ndtvadsrqf("fozwvlovdui")` — **prefix match**, includes red and blue. L7 initial count = 3.
- Only way to decrement: line 5397, `ddaguepwkt -= 1` when middle cell has a piece with **exactly** matching `.name` as the jumper (`qcerbdpdcl.name == uywtlohliu.name`). Since there's 1 red and 2 non-blue Ns, only N-over-N removal reduces count to 2.
- Win is only ever triggered via `cfilhtifcb → tdcblgbfxw → self.win()`. ACTION5 (`kuexigxyxw`) only calls `pchvqimdvj` (visual confetti + spawns `cwyrzsciwms` button). ACTION6 click on the button at x<16,y>48 only queues render frames — does **not** call `win()`.
- Jump validity (`qikmikecdf`): middle cell must contain `fozwvlovdui*` OR `dgxfozncuiz` (peg) — peg-jumps are real. Landing must satisfy `posalhhmjq`: len(objs)==1 and "hupkpseyuim" in name, OR len(objs)==2 and "hupkpseyuim2" in names.
- Scroll triggers for L7 in `cfilhtifcb` (lines 5493–5511) at landings `(8,8)`/`col==16` only animate `hncnfaqaddg.cdpcbbnfdp` (pixel offset). Grid coordinates (`porbskbertu`, `abvcoxnskr`) are **not** modified. `pneghtfqtt` (the one function that would sync pixel→grid) is defined but never called. Scroll is visual-only.
- Right-N @(22,6) neighbors: (22,5)=walkable; (21,6)/(22,7)/(23,6)=empty/OOB. Middle (22,5) has no piece/peg → `qikmikecdf` returns False for all 4 directions. Right-N gets zero direction arrows. Confirmed by engine probe: no push sequence (tested to depth 6) unlocks a valid direction for right-N.
- Block at (22,4) is `;` = `[hupkpseyuim2, kraubslpehi-up]` — no peg, not jumpable-middle. Other blocks in L7: `(5,5), (14,2), (18,3), (19,2), (20,3), (22,4)` — none adjacent to right-N, none movable into positions adjacent to right-N.

**Geometric impossibility argument**: For N-over-N removal, left-N must reach a cell where right-N is the middle of a valid jump. Required landing cells: (22,4) valid; (22,8), (20,6), (24,6) invalid (empty/OOB). The only viable self-jump would need left-N at (22,8) jumping up over (22,6) to land at (22,4) — but (22,8) is off-grid (row 8 ends at col 12). Or left-N at (22,4) jumping down to (22,8) — landing invalid. **There is no valid self-jump configuration that eliminates right-N.**

**What was ruled out**:
- Pegs moving with pushed blocks (this IS correctly modeled in solver).
- Hidden state changes via ACTION6 click on empty cells / pegs / walls (traced every branch of `dghsidbuet`).
- Scroll-induced grid mutation (scroll is pure pixel animation; `pneghtfqtt` is dead code).
- `cwyrzsciwms` button triggering `win()` (it does not; only schedules render frames).
- ACTION5 `kuexigxyxw` triggering `win()` (it only calls `pchvqimdvj` animation).
- Piece removal outside `cfilhtifcb` (only one call site to `faretcdgmc` with a piece, at line 5400).

**What remains unexplained**: The task description asserts (based on previous investigation) that L7 must have an unmodeled mechanic. I could not find it. Two possibilities:
1. The mechanic exists but I missed it (most likely candidates: some interaction in `cfilhtifcb`'s post-move scroll branches I didn't trace through; or an interaction between red piece and N's that I missed).
2. L7 is truly unwinnable as-coded in this engine version, and the game-level "unsolvable" state is known to the game authors (unlikely but possible).
