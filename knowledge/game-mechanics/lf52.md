# lf52 -- Grid-based puzzle with tile movement, merging, and custom rendering engine

## Toolset
Available actions: [ACTION1 (UP), ACTION2 (DOWN), ACTION3 (LEFT), ACTION4 (RIGHT), ACTION6 (CLICK), ACTION7 (UNDO)]
This game uses a custom internal scene graph (`rjfmjxejeiq`) rather than standard ARC sprite positioning. It has its own animation framework, camera scrolling, and tile-based grid. UNDO is available when `STORES_UNDO = True`.

## Objects
The game uses a scene graph with named nodes rather than sprite tags. Key object types:
| Name pattern | Role |
|---|---|
| `fozwvlovdui` | Primary game tiles -- clickable, movable puzzle pieces |
| `fozwvlovdui_red` | Red variant tiles |
| `fozwvlovdui_blue` | Blue variant tiles (static in levels 8+) |
| `lgbyiaitpdi` | Direction indicator arrows (4 cardinal directions) |
| `hupkpseyuim` / `hupkpseyuim2` | Grid cell markers -- define the playable grid |
| `kraubslpehi` | Walls/boundaries within the grid |
| `cwyrzsciwms` | Special clickable trigger (activates `zvcnglshzcx` flag) |
| `bhdfjlqapap_0/1` | Tutorial/hint animation sprites |
| `csrvckunbev` | Player cursor/selector |

Color palette: BACKGROUND_COLOR = 0 (LIGHT_BLUE mapped to 0), custom rendering via scene graph.

Camera: `Camera(background=0)` with custom `wnrtmoelug` interface for frame rendering. The game renders frames via `equnaohchtj.render()` and passes them to the camera interface.

## Interaction Model
- **ACTION1 (UP)**: Increments step counter. Calls `tmhxwcojkh(0, -1)` -- moves `hupkpseyuim2` tiles upward. Tiles are sorted by grid_y (descending for up). If a tile would move into a `kraubslpehi` wall, it stops. If it meets another `hupkpseyuim2` tile, it passes through. Moving a `fozwvlovdui` tile into a wall may trigger camera scrolling (`hncnfaqaddg.cdpcbbnfdp` adjustment) with animated panning, varying by level.

- **ACTION2 (DOWN)**: Same as UP but `tmhxwcojkh(0, 1)`.
- **ACTION3 (LEFT)**: `tmhxwcojkh(-1, 0)`.
- **ACTION4 (RIGHT)**: `tmhxwcojkh(1, 0)`.

- **ACTION5** (mapped but calls `kuexigxyxw` which just calls `pchvqimdvj()` then passes -- likely a "confirm" or no-op in current implementation).

- **ACTION6 (CLICK)**: Converts click coordinates to scene graph position via `rozmjeilxc(x, y)`. Behavior depends on what's clicked:
  - Clicking a `fozwvlovdui` tile: selects it and shows direction arrows (`lgbyiaitpdi`) for valid movement directions.
  - Clicking a `lgbyiaitpdi` arrow: moves the selected tile in that direction. The tile slides to a new grid position, and if a same-colored `fozwvlovdui` tile is in the path, they merge (the intermediate tile is removed with a shrink animation).
  - Clicking `cwyrzsciwms`: activates a special undo/reset button that appears in the bottom-left corner.
  - Clicking the bottom-left area when `zvcnglshzcx` is active: triggers a full level reset animation.
  - Clicking empty space: clears the selection/arrows.

- **ACTION7 (UNDO)**: Calls `aphzlzlgni()` which restores the previous state from the undo stack. Increments step counter.

- **Animation system**: Fully custom. The game builds frame lists (`list[np.ndarray]`) and passes them to the camera interface. The `step()` method checks `xcjqviblhlt` flag -- on first call it processes the action and generates frames, on subsequent calls it drains the frame queue until animation completes.

## Peg-solitaire ground truth (decoded April 2026, CBP)

The surface appearance of "tile merging" is actually **peg-solitaire with a push-transport layer**.

- Pieces: `fozwvlovdui` (neutral/N), `fozwvlovdui_red` (red/R), `fozwvlovdui_blue` (blue/B).
- Static pegs: `dgxfozncuiz`.
- Pushable blocks: `hupkpseyuim2` — slide through adjacent `kraubslpehi*` wall cells when ACTION1-4 is pressed. Multiple blocks move in one push (sorted in push direction).
- A piece riding on a block (cells where block is one of the objects) is carried by the push.
- `dgxfozncuiz` on a block-cell (`P`/`D` glyphs) also rides the push.
- `hupkpseyuim` (walkable tiles) do NOT move with blocks in practice — blocks in this game always spawn on wall cells, never on walkable-only cells.

**Jump rule** (`qikmikecdf`): selecting a piece and clicking an arrow fires a 2-cell jump. Middle cell must contain any `fozwvlovdui*` OR `dgxfozncuiz` (`pymqmlkgzs`). Landing cell must be `len==1 and name contains hupkpseyuim` OR `len==2 and one is hupkpseyuim2` (`posalhhmjq`). A piece on a landing cell makes it invalid.

**Removal rule** (`cfilhtifcb` line 5393-5396): after a jump, if the middle cell contains a non-blue `fozwvlovdui*` piece whose `.name` matches the jumper's `.name`, that middle piece is removed. Blue middles are never removed; different-type matches (N vs R) are never removed.

**Blue is movable, not static.** This was the April 2026 model bug: prior solver excluded blue from `get_valid_jumps`. Blue pieces CAN be clicked, CAN be moved (via any direction where `qikmikecdf` holds), and — as either jumper or middle — participate in all landing/jumpable-middle checks. Blue moves just don't change the win counter. Once this bug was fixed, L8 and L9 collapsed from "0 reducing jumps anywhere in the reachable push+jump space" to solvable in seconds, because blue pieces often sit between two neutrals and are the only jumpable middle in the path.

**Win counter** (`ddaguepwkt`): `len(ndtvadsrqf("fozwvlovdui"))` — prefix-match includes N+R+B. For levels ≥ 8, subtract `self.lzoqlpcwzpu` (initial blue count, which doesn't change since blues can't be removed in normal play with no matching jumps). Win triggers when counter reaches 1 (or 2 for L6/L7) after a reducing jump at line 5572.

**L6 special win** (line 5472-5474): clicking the neutral piece at grid (16,2) while red is parked at (6,6) triggers `pchvqimdvj()` — a position-based win that doesn't require a jump. L7 has no analogous position-based win trigger; its win is purely counter-based.

## Win Condition
The `equnaohchtj` scene graph sets `iajuzrgttrv = True` via its `win()` method. This is checked after animation completes:
```python
if self.ikhhdzfmarl.iajuzrgttrv:
    self.next_level()
```
The win condition is triggered internally by the scene graph when all `fozwvlovdui` tiles have been merged/cleared from the grid (or a target configuration is reached). The exact trigger is inside `pchvqimdvj()` and level-specific position checks.

**Lose conditions**:
- `evxflhofing` flag set (explicit lose state)
- Level 1: step counter (`asqvqzpfdi`) >= 64
- Levels 2-5: step counter >= 64 * 5 = 320
- Levels 6+: step counter >= 64 * 10 = 640

## Level Progression
10 levels. Grid cell size is 6 pixels. Changes:
- Level 1: Small grid (10x5 offset), tutorial hints appear, simple merge puzzle
- Level 2: 6x8 grid offset
- Level 3: 5x5 grid offset, camera scrolling on tile push
- Level 4-5: Directional push variations
- Level 6-7: More complex scrolling mechanics
- Level 8+: Blue tiles introduced (static, counted separately)
- Level 9-10: Full complexity with combined mechanics

Step limits scale with level: L1 has 64-step limit, later levels allow 320 or 640 steps.

- CLICK-based selection + directional movement. Click tile, then click arrow to move it.
- Directional keys move ALL `hupkpseyuim2` tiles simultaneously in that direction.

## 2026-04-12 Empirical L7 Probe — New Findings

## L7 Final — Path-Dependency Probe (2026-04-12 session 6)

**Hypothesis tested**: L7's initial state might depend on HOW it is
reached (sequential play vs direct set_level), via residual cross-level
state that no previous agent had explicitly compared. This was the one
assumption five prior agents shared but never tested.

**Experiment**: `arc-agi-3/experiments/lf52_l7_pathdep.py` builds two
fresh `Lf52` instances, drives one through the solver's L1..L6 cached
solutions (Path B), loads the other directly with `game.set_level(6)`
(Path A), then dumps every attribute of the `Lf52` wrapper, the `eq`
(`ikhhdzfmarl`), the full scene graph (recursive walk of sprites, pegs,
pieces, blocks, walls, walkable cells, fixed pegs, pixels), the rendered
frame hash, and the numpy RNG state. A field-by-field diff is then taken.

**Result**: Board state **IDENTICAL** between Path A and Path B.

Total fields differing between the two snapshots: 15. Every differing
field is pure bookkeeping or cosmetic:
- `_full_reset` (True for fresh, False after play)
- `_score` (0 vs 6 — expected, Path B completed 6 levels)
- `foztuzaztfm[0..8]` hashes (win-animation mask arrays; computed once
  at `Lf52.__init__` from `vjhacvjubl()` which calls `np.random.shuffle`.
  Differs because global `np.random` state evolved during Path A's
  execution before Path B's `arcade.make()` consumed it — an inter-run
  artifact, NOT within-game state)
- `np_rng[1..2]` (global numpy RNG state — same cause)
- `frame_hash` / `frame_sum` (driven by `foztuzaztfm`-colored cells)

No piece positions, no block positions, no fixed peg positions, no wall
positions, no walkable cells, no camera offset, no step counter, no
engine flags, no scene-graph sprite differs between the two paths.

**Conclusion**: Path dependency is **ruled out**. L7 is produced by a
fresh `equnaohchtj()` via `on_set_level()` (lf52.py:5864) regardless of
how L7 was reached. The `next_level()` pathway (base_game.py:412) goes
through `_really_set_next_level()` → `set_level(idx+1)` → the exact
same `on_set_level` hook as direct loading. The engine leaves no
residual state that L7's `qjwmwkhrml()` can read.

**What this closes**: the sixth and final open question about L7 — the
one nobody had tested. Combined with sessions 1-5 (push mechanics,
click registration, counter semantics, `win()` call sites, right-N@(22,6)
