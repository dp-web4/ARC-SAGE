# vc33 — Seesaw-balance puzzle with sliding beams and colored indicators

## Toolset
Available actions: [ACTION6 (click only)]. Pure click game. Clicking triggers beam operations or rotation swaps. Implies a spatial reasoning / physics-style puzzle.

## Objects
- `rDn` (tag) — Rectangular blocks/beams of various sizes (e.g., 14x10, 12x9, 14x48). Static structural elements that form the seesaw arms. Color 0 (black).
- `UXg` (tag) — Vertical bar sprites (e.g., BqR is 3px wide, 54px tall with a gap). These are pivot/fulcrum markers between beam segments.
- `ZGd` (tag, also `ACQ`) — Clickable trigger sprites. Clicking these activates the seesaw transfer operation (shrink one side, grow the other).
- `zHk` (tag, also `ACQ`) — Clickable rotation triggers. Clicking these initiates a rotation animation that swaps indicators between beam ends.
- `fZK` (tag) — Small colored indicator sprites (e.g., 2x3 with color 14). These represent the "weights" or markers that must be positioned correctly.
- `HQB` (tag) — Target/goal indicator sprites. These sit on beams and must end up at the correct positions.
- `WuO` (tag) — Boundary/wall sprites that limit beam movement.
- `rlV` (tag) — Visual indicators (added/removed dynamically) showing locked/unlocked state of rotation triggers.
- `KEA`, `HPB` — Lock indicator sprites spawned when a rotation trigger becomes ready.

## Interaction Model
- **CLICK (ACTION6):** Uses `display_to_grid` for coordinate mapping. Two click targets:
  1. **ZGd sprites (seesaw triggers):** Calls `ccl()` which calls `gel()`. This transfers length from one beam to an adjacent beam -- one shrinks by removing pixel rows, the other grows by adding rows. The HQB indicators on those beams move accordingly. Like a seesaw: push one side down, other goes up.
  2. **zHk sprites (rotation triggers):** Only active when `krt()` returns true (beam is properly flanked by adjacent beams). Calls `teu()` which creates a multi-step animation (`ysn` sequence) that swaps HQB indicators between the two beam ends, and bounces the trigger sprite.
- **Animation system:** Uses `ysn` (sequence of `diu` movement steps). Each `diu` moves a sprite 1px per frame toward a target position. Sequences chain together. Game blocks during animation.

## Win Condition
Method `gug()`: For each `HQB` sprite, get its color from `pixels[-1,-1]`. Find which `rDn` beam it sits on. Then check all `fZK` indicators: there must exist an fZK that (a) has the same color as the HQB, (b) is at the same cross-axis position, and (c) sits on a `UXg` pivot adjacent to that beam. In plain English: **every target indicator (HQB) must have a matching-color fZK indicator placed on the correct adjacent pivot**. All must be satisfied to win.

## Level Progression
**7 levels.** Grid sizes: 32x32 (L1-2), 52x52 (L3), 64x64 (L4-7).
- `TiD` data controls direction: `[2,0]`=right, `[-2,0]`=left, `[0,2]`=down, `[0,3]`=down-fast, `[3,0]`=right-fast, `[-3,0]`=left-fast, `[0,-2]`=up.
- `RoA` = step counter (energy). 50 for small levels, 200 for large.
- Level 1-2: Single orientation, few beams. Tutorial levels.
- Level 3-4: Larger grids, more beams and pivots, orthogonal direction.
- Level 5-7: Full 64x64 grids with complex beam networks, multiple seesaw chains, rotation needed.

All sprites in a level share the same rotation value matching the direction (0/90/180/270).

## No-Source Discovery Strategy
1. **First click:** Click a visible colored trigger element (ZGd). Observe one beam shrinks, another grows.
2. **Observe:** Small colored indicators move with the beams. Notice lock symbols appear on rotation triggers.
3. **Hypothesis:** This is a seesaw/balance game. Transferring beam length moves indicators, and rotation swaps indicators between ends.
4. **Test rotation:** Click an unlocked rotation trigger (zHk). Watch indicators swap sides.
5. **Goal recognition:** Match each target indicator with the correct adjacent colored marker. Once all match, level completes.

## Key Patterns
- Direction tuple `TiD` is crucial -- determines which axis beams slide along and the step size per click.
- The `lia()` method checks if horizontal movement (oro[0] > 0 or oro[1] > 0) vs vertical.
- Coordinate helpers swap x/y based on direction, so the same logic works for all orientations.
- `gel()` transfers exactly `abs(TiD[0])` or `abs(TiD[1])` pixels per click from one beam to another.
- Rotation is only available when a trigger sprite has beams properly flanking it on both sides (`krt()` check).
- Energy bar rendered at row 0 (7=orange when filled, 4=yellow when empty). Each click costs 1 energy via `czh()`.
- Lose condition: energy reaches 0 (`self.vrr.olv` becomes falsy).
- Colors: background=3 (green), padding=4 (yellow), beams=0 (black), pivots=5 (grey), indicators=14 (grey), targets vary.
