# bp35 -- Side-scrolling platformer with gravity, tile grid, and gem collection

## Toolset
Available actions: [3, 4, 6, 7] (LEFT, RIGHT, CLICK, UNDO).
Note: Actions 1, 2, 5 exist in the dispatch but are NOT in `available_actions`. This is a 2D side-scroller where the player moves left/right and gravity handles vertical movement. CLICK interacts with clickable tiles. UNDO reverses to previous state.

## Objects
The game uses a custom scene graph engine (`yodvybvftxa`) with its own sprite/animation system, NOT the standard ARC sprite system directly. Levels are built from ASCII grid maps with 6x6 pixel tiles.

Camera: Custom `mrmwmwiksi` class with position tracking (`rczgvgfsfb`). Camera Y follows the player: `player.grid_y * 6 - 31 - 5`.

## Interaction Model
- **ACTION3 (LEFT)**: Move player one grid cell left. Player sprite flips to left-facing. Triggers gravity fall check.
- **ACTION4 (RIGHT)**: Move player one grid cell right. Player sprite flips to right-facing. Triggers gravity fall check.
- **ACTION6 (CLICK)**: Converts screen coordinates to grid position via `hyntnfvpgl(x, y + camera_offset)`. If clicking a `qclfkhjnaac` (ground) tile, triggers a 3-frame destroy animation. This removes that tile, potentially causing the player to fall.
- **ACTION7 (UNDO)**: Restores previous state snapshot (full scene graph rollback via `waummvnxwvp` state manager). Resets animation frames.
- **RESET**: Full level reset via `eubgwokpez()`.

Gravity system (`fsvnqdbzrp`):
- Direction depends on `vivnprldht` (True = gravity points up/dy=-1, False = gravity points down/dy=+1).
- After each horizontal move, player checks the cell in gravity direction.
- If that cell is empty/passthrough, player falls until hitting solid ground, gem, or spikes.
- Fall distance determines animation frame count (`iwuohihxyl = min(20, 3 * fall_distance)`).
- Landing on gem (`fjlzdjxhant`) -> win. Landing on spikes (`ubhhgljbnpu`/`hzusueifitk`) -> lose.

Moving platforms (`mylefxfaev`): On levels 1-3, every other move triggers platforms `aknlbboysnc` and `jcyhkseuorf` to move up by one cell. If player Y matches platform Y after move, player is crushed -> lose.

Animation system: Uses `urkteguwnke` sequences (move, wait, callback) rendered as multi-frame blocking animations. The `step()` method uses a two-phase approach: phase 1 dispatches action and queues animation, phase 2 waits for animation to drain before resolving win/lose.

State snapshots: `vlyikbzinq()` saves full scene state before each move. `svmaaixutx()` restores from last snapshot. History includes scene graph node positions, properties, and tree structure.

## Win Condition
Method `win()` on `yodvybvftxa`: sets `nkuphphdgrp = True`.
Triggered when player lands on `fjlzdjxhant` (gem) tile, or walks into it horizontally.

In `Bp35.step()`: after animation drains, checks `oztjzzyqoek.nkuphphdgrp` -> `next_level()`.

Loss: `jrhqdvdwpsb = True`, set when landing on spikes or crushed by moving platform.

## Level Progression
9 levels (grid1 through grid9, indexed 1-9). Each level is an ASCII grid map with:
- Tile size: 6x6 pixels per grid cell
- Varying width/height
- Progressive complexity: more gaps, more spikes, moving platforms, clickable-tile puzzles
- Gravity direction may vary (`vivnprldht`)
- Levels 1-3 have moving platforms that shift each other move

## No-Source Discovery Strategy
1. Press LEFT/RIGHT to confirm horizontal movement and observe character walking.
2. Notice the character falls when walking off a ledge -- gravity is automatic.
3. Look for colored tiles below: spikes (deadly), gems (goal).
4. Try CLICK on a ground tile to see it disappear -- this is the core puzzle mechanic.
5. Try UNDO to restore removed tiles.
6. Observe camera scrolling vertically.
7. Check if platforms move after you move.
8-10. Map the level layout by moving and clicking strategically. Plan a path to the gem.

## Key Patterns
- This is a puzzle-platformer, NOT a standard platformer. The key mechanic is CLICKING to destroy ground tiles to create paths.
- Player only moves horizontally; vertical movement is entirely gravity-driven.
- Destroying the wrong tile can make the level unsolvable -- use UNDO liberally.
- Moving platforms on early levels add a timing element (they move every other player move).
- The camera only shows a vertical slice; mental mapping of the full level is important.
- Gem position is the goal; plan a path that lets you fall onto it.
- Spike tiles instantly kill; avoid creating fall paths onto them.
- State snapshots enable full undo, so experiment freely.

## Viewport/Camera Constraint (viewport-aware solve)

The API enforces that CLICK (x, y) must lie in [0, 63]. World-to-viewport:
`viewport_y = world_y * 6 - camera.y`. Camera updates only on falls:
- grav UP: `camera.y = fallen_y * 6 - 36`
- grav DN: `camera.y = fallen_y * 6 - 26`

Walks (L/R) that don't trigger a fall do NOT scroll the camera. Consequence:
to click a tile outside the current viewport, the player must first reach (via
a fall) a position that brings the target into view. This motivated the
"walk-trigger-fall-then-click-in-view" pattern used in L3, L4, L7, L8 rewrites.

### Tile interactions (additional)
- `etlsaqqtjvn` (E): clickable — click SPREADS E to all empty 4-neighbors and
  removes the clicked E. Used as dynamic platforms / fall stops.
- `yuuqpmlxorv` (B): clickable — click converts B to `oonshderxef` (O, passthrough).
  O is walkable and allows fall-through.
- `lrpkmzabbfa` (G): clickable — click flips `vivnprldht` (gravity) AND removes
  the G block. Player falls/bounces immediately based on new gravity direction.
  After N G-clicks, gravity flipped N times.
- `qclfkhjnaac` (D): clickable — click destroys the tile. Triggers fall for
  player if D was adjacent in gravity direction.

### Structural OOB limits (updated 2026-04-13 second pass)

Only **one** OOB click remains after second-pass investigation:

- **L5 C(4,31)**: STRUCTURAL. The gem lives in an enclosed sub-chamber
  (cols 1-3 at y=30-31), sealed by walls at y=29 cols 0-8 and y=32 whole row.
  Col 8 is the only fall-access to the bottom chamber; player lands at
  (8,31), from which horizontal L-walk blocks at G(4,31). Clicking G(4,31)
  flips gravity UP; fall-up from cols 5-8 stops at y=30 due to wall at
  (X,29) for X∈[1,8]. From (X,30) the wall at (4,30) blocks L-walk to gem
  sub-chamber. Third G-block (8,1) is fully enclosed by walls, unreachable.
  No E-seeds present on L5. Walls are non-destroyable. No path exists.

- **L7 C(5,2)** — PREVIOUSLY flagged OOB, now UNBLOCKED. The first-pass
  verdict that the spike ring at y=6-10 made y≤8 unreachable was wrong.
  The ring has a gap at y=7 cols 4-6 (walls cols 0-3 and 7-10; spikes at
  y=6 only in cols 4-6, cols 0-3 and 7-10 at y=6 are walls). E-spread
  seeded from E(3,18) can reach this gap via iterated col-6 click-and-fall:
  each C(6,Y) spreads to (6,Y-1) and re-creates E(6,Y+1) as a stop for the
  next fall-up. 9 clicks walk player from (6,17) up to (6,8), at which
  point cam_y=12 puts G(5,2) vy=0 (in view). Cost: 13 extra actions over
  the old 1-OOB solve, but 0 OOB. Path shipped in solver; L7 now 48 actions.

Teaching-grade lesson: a "structural" verdict based only on "player can't
reach the click spot with current grav+walk primitives" ignores the
E-spread primitive as a state-modifier that CREATES new fall-stops along
the path. The framing trap is treating the board as static when E turns
empties into solids en route.
