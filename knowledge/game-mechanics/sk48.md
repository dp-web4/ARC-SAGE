# sk48 — Rail Weaver: Extend Colored Rails to Match Target Patterns

## Toolset
Available actions: [ACTION1=UP, ACTION2=DOWN, ACTION3=LEFT, ACTION4=RIGHT, ACTION6=CLICK, ACTION7=UNDO].
Arrows extend/retract the selected rail. Click switches between rail pairs. Undo reverts to previous state. This is a pattern-matching rail puzzle.

## Objects
- **Rail heads** (tag `epdquznwmq`): 6x6 colored blocks with inner 4x4 detail and transparent corners. Colors: `ejlpqgojjt` (green/6 border, yellow/4 inner), `udbuodqlxv` (white/15), `xtuqlbebvk` (grey/5 with orange/10 center), `zkekdulqku` (grey/5 with cyan/11 center). These are the "engines" that push rail segments. Appear in pairs (one above the board, one below).
- **Rail segments** (`qtjqovumxf`): 6x6 tiles with horizontal stripe pattern (colors 2/green and 3/dark-green). Can be oriented horizontally (rotation 0/180) or vertically (rotation 90/270). These extend from rail heads.
- **Target color blocks** (`elmjchdqcn`): 6x6 blocks, color 8 (grey) default, remapped to target colors (8/grey, 9/maroon, 12/blue, 14/yellow). These sit on the bottom row and define the pattern to match.
- **Track/rails** (`irkeobngyh`): 2x8 vertical bars (colors 2/3 stripe). These are the physical tracks that rail segments slide along. Not interactable.
- **Board background** (`jtteddgeyl`): Solid color 4 (yellow) rectangles defining the play area. Tagged for bounds checking.
- **Walls** (`mkgqjopcjn`): 6x6 color 5 (grey) blocks that block rail movement.
- **Match indicators** (`kevthtkmzm`): 4x4 blocks with color 0 (black) center. Appear between paired rail segments to show matches. Visible when colors match, hidden when they don't.
- **Crush indicator** (`pkzxknabii`): 6x6 frame showing a segment about to be crushed (pushed off the board).
- **Separator line** (`yukipuenar`): Full-width line of color 3 at row 53, dividing upper board from lower staging area. Removed at level start.
- **Step counter UI**: Progress bar on row 53 (the separator line position). Color 2 (filled) vs 3 (empty). 196 steps per level.

Constants: `udenqlsrfq = 6` (tile size), `fzjeqdahvs = 53` (separator Y), `hhvuoijeua` = direction map {0:(1,0), 90:(0,1), 180:(-1,0), 270:(0,-1)}.

## Interaction Model
- **CLICK (ACTION6)**: Selects a rail head pair. Each head in the upper area is paired with one in the lower area (`xpmcmtbcv` mapping) based on matching inner color. Selecting highlights the pair (color remap to active colors).
- **Arrows (ACTION1-4)**: Each arrow press moves the selected rail:
  - **Along rail direction** (matching head's rotation): extends the rail by adding a new segment at the head position and pushing all existing segments forward by one tile (6 pixels).
  - **Against rail direction**: retracts by removing the closest segment to the head.
  - **Perpendicular with track support**: if there's an `irkeobngyh` track adjacent in the movement direction, slides the entire rail and head perpendicular.
  - Segments can push colored target blocks (`elmjchdqcn`) when they collide.
  - Segments can be "crushed" when pushed off the board edge.
- **UNDO (ACTION7)**: Restores the previous snapshot state (positions and segment counts).
- **Collision/bounds**: Movement blocked by board edges (`jtteddgeyl` bounds) and wall blocks (`mkgqjopcjn`).
- After every move, `gvtmoopqgy()` checks if rail segments on paired rails have matching colors in corresponding positions. Match indicators become visible for matching pairs.

## Win Condition
Method `gvtmoopqgy()`: For each paired rail head (upper + lower), compare the color at position `[1,1]` of each segment's `elmjchdqcn` block that overlaps with that rail segment. **Win = every paired position has matching colors** between the upper and lower rails, with indicators visible for all pairs. Returns True when all match.

## Level Progression
**8 levels**, all on 64x64 grid. Changes:
- **Level 1**: 2 rail heads, 3 target blocks below + 3 above. Simple horizontal rails.
- **Level 2**: 2 rail heads, 5 target blocks below + 5 above. Longer rails.
- **Level 3**: 2 rail heads, some targets on vertical rail (rotation 90). Mixed orientations.
- **Level 4**: 3 rail head pairs (with `xtuqlbebvk` and `zkekdulqku` variants). Vertical and horizontal rails.
- **Level 5**: 2 rail heads, wall block (`mkgqjopcjn`) obstructing. Must navigate around obstacles.
- **Level 6**: 2 rail heads, wall block, more target blocks. Track layout with horizontal/vertical crossings.
- **Level 7**: 2 rail heads, complex multi-color target arrangement.
- **Level 8**: 2 rail heads, shorter rails, fewer targets, more constrained.

Key data per level: `grouped_pauses` (bool), `lit_extension` (bool).

## No-Source Discovery Strategy
1. **First action**: Press an arrow key. Observe a colored rail segment appears/extends from the highlighted rail head.
2. **Observe**: Note colored blocks on the bottom staging area and on the board. Note the small indicators between rails that light up or stay dark.
3. **Hypothesis**: The goal is to make the upper rail's colored pattern match the lower rail's pattern. Indicators show which positions already match.
4. **Test**: Extend the rail so segments overlap with target color blocks. Check if indicators light up when colors align.
5. **When ready**: Plan rail extension to align all colors between paired rails. Use perpendicular movement (via tracks) and undo as needed.

## Key Patterns
- Tile size is 6x6 pixels. All positions are multiples of 6.
- Each rail head has a direction based on rotation. Extension adds segments in that direction.
- Paired rails (upper/lower) must have matching colored overlaps at each corresponding position.
- 196-step limit per level (generous).
- Undo saves full state snapshots -- use liberally.
- Perpendicular sliding requires an `irkeobngyh` track sprite at the adjacent position.
- Segments can push `elmjchdqcn` blocks and even get "crushed" at board edges.
- Win triggers a 35-frame celebration animation (39 frames observed in practice) before advancing.

## CBP L2+ Findings (2026-04-12)

### WIN CONDITION CORRECTION
The L1 "visit all targets" finding was CORRECT for L1 but NOT the general mechanism. The actual win condition (gvtmoopqgy in source) compares target colors at paired rail segment positions **INDEX by INDEX**. vjfbwggsd[head] is built by scanning segments left-to-right, collecting any target at each (x,y). Win = all indices match in color between upper and lower rails. Targets don't need to be physically moved — they just need to BE at segment positions with the correct color ORDER.

### PERPENDICULAR PINNING
Targets at positions with a perpendicular rail segment CANNOT be pushed in the extension direction. The rail passes THROUGH, coexisting at the same position. The target stays fixed. This applies to both extension (push RIGHT) and retraction (pull LEFT). To free a pinned target: push it PARALLEL to the pinning rail via slides.

### L2 STRATEGY (Phase 1-3 complete, Phase 4 blocked)
L2 has a vertical rail at x=29 y=0-24 that pins all upper targets. The solution requires:
1. **Push stack down** (13 moves): slide from y=6 with 5 segs, push 3 times
2. **Clear (29,30)** (5 moves): extend to push target(9) to x=35
3. **Free target(14)** (4 moves): slide from y=18 with 5 segs pushes target(14) from (29,24) to (29,30)
4. **Arrange** (unsolved): place all 4 targets at distinct x on same row in order [8,12,9,14]

The arrangement step is blocked by chain-push collisions. The extend-to-edge technique works for COLLECTING targets, but ordering them requires careful sequencing. The first collected target ends up leftmost after retraction.

## Waving-Cat L1 Findings (2026-04-11)

### PALETTE WARNING
The color names in this document are WRONG for the current game rendering. The doc says `elmjchdqcn` target colors are 8/grey, 9/maroon, 12/blue, 14/yellow. **Actual rendered colors (verified against human game UI): 8=RED, 9=BLUE, 14=GREEN.** The doc was likely written under the pre-2026-04-06 palette that was fixed in dp-web4/sage-arc-collab#1. The static reference screenshot `game-screenshots/sk48_L1.png` also uses the old palette. **Do not trust color NAMES in this doc — use visual perception.**

### NON-SOLID TARGETS (Critical for solving)
**Targets (`elmjchdqcn`) are NOT solid to the rail.** When the rail extends so a segment sprite overlaps a target sprite (same row + column range), the rail PASSES THROUGH the target. The target becomes embedded in the rail's stripe pattern — the target's 4-wide colored area sits between the segment's 2-wide stripe edges. This is the core mechanic of sk48.

Visual signature of target embed: **3-frame animation** (vs normal 2-frame extension) and **reduced pixel change** (4-9px vs normal 12px). Gemma should learn to recognize this signature as "target interaction occurred."

### SLIDE BEHAVIOR DEPENDS ON RAIL LENGTH
- **Short rail** (1-3 segments, not reaching target column ~cols 41-46): slides freely without pushing targets. Use retraction to shorten before sliding.
- **Long rail** (5 segments, reaching target column): UP slides push any vertically-adjacent target stack. DOWN slides never push targets (nothing below the rail to push).
- This asymmetry enables **decoupling**: slide UP (push stack), slide DOWN (rail moves alone, stack stays). But for basic solving, just retract to short rail before sliding.

### WIN CONDITION (L1 verified)
The win triggers when **all upper targets have been "visited"** (the rail has extended through each target at least once at that target's row). The win does NOT require targets to be at specific positions or the rail to be at a specific row at the moment of win. L1 win fired on the final extension that visited the last unvisited target (BLUE), producing a 39-frame/1186px celebration animation.

**Clean solve sequence for L1** (~21 actions from pristine):
1. Slide UP to RED's row (head stripes at rows 20-21 within RED rows 19-22)
2. Extend RIGHT x4 (rail passes through RED at P5 = visit)
3. Retract LEFT x4 (shorten rail to 1 segment for clean slide)
4. Slide DOWN to GREEN's row (head stripes at rows 32-33 within GREEN rows 31-34)
5. Extend RIGHT x4 (visit GREEN)
6. Retract LEFT x3 (shorten to 2 segments)
7. Slide UP to BLUE's row (head stripes at rows 26-27 within BLUE rows 25-28)
8. Extend RIGHT x3 (visit BLUE = WIN, 39-frame celebration)

Note: the number of slides needed depends on the decoupling state. With a short rail, slides don't push targets so you can navigate freely between rows. Reaching each target requires extending to the board's right edge (~5 segments = 4 extensions from 1).

### SOLVER SCAFFOLD BUG
`game-solvers/sk48_solver.py` emits click actions as `f"click({cx},{cy})"` strings — a coordinate-based format. But ACTION6 is a pair-cycle toggle with NO coordinate. The scaffold's click output is silently ignored by the game engine. Proposed fix: replace `f"click({cx},{cy})"` with `"ACTION6"`. See `SAGE/arc-agi-3/waving-cat/sk48_solver_click_fix.md` for full analysis.
