# ft09 — Constraint-satisfaction coloring puzzle with neighbor rules

## Toolset
Available actions: [ACTION6 (click only)]. Pure click game. Clicking a cell toggles its color through a palette cycle. The constraint pattern determines which neighbors are affected. This is a logic/coloring puzzle akin to a lights-out variant with color cycling.

## Objects
- `Hkx` (tag) — Standard grid cells (clickable). Small squares placed on a 4px grid. Colored with the active palette. Clicking cycles the color of affected neighbors.
- `NTi` (tag) — Pattern cells (clickable). 3x3 sprites with color-6 dots marking the influence pattern. Clicking uses the cell's own dot pattern instead of the global pattern.
- `bsT` (tag) — Constraint/rule sprites (3x3). Each has a unique multi-color pattern encoding a neighbor constraint. The center pixel color is the "required" color; surrounding pixels encode which neighbors must match (color 0) or differ (non-0) from that center color.
- `cgN` — Large 16x16 background block (color 5, grey). Static backdrop.
- `dvQ` — Decorative/frame sprite.
- `bzH` — Diamond-dot grid overlay (5x9, color 6 dots). Visual reference.
- `Ycb` (tag) — Tutorial indicator sprite (level 1 only). Flashes when clicking empty space, hinting you should click cells.
- Various info sprites per level (step counter display, etc.).

## Interaction Model
- **CLICK (ACTION6):** Uses `display_to_grid` for coordinate mapping.
  1. **Click an Hkx cell:** Applies the global influence pattern (`irw` / `elp` from level data) to determine which neighbors are affected. For each position (i,j) where the pattern has value 1, the cell at offset (i-1, j-1)*4 from the clicked cell has its color cycled to the next in the palette.
  2. **Click an NTi cell:** Same as above, but uses the NTi sprite's own dot pattern (positions with color 6 = active) instead of the global pattern.
  3. **Click empty space (level 1):** Triggers a 4-frame flash animation on the tutorial indicator.
  4. **Click a bsT constraint sprite:** Nothing happens (no action taken, but avoids the empty-space flash).
- **Color cycling:** `gqb` (from level data `cwU`) defines the color palette as a list (e.g., [9, 8]). Each affected cell cycles: find current color index in palette, advance by 1, wrap around. With 2 colors, it's a toggle. With 3 colors, it cycles through all three.

## Win Condition
Method `cgj()`: For every `bsT` constraint sprite, check all 8 neighbors (3x3 grid minus center). The constraint sprite's center pixel `[1][1]` is the "required color". For each of the 8 neighbor positions:
- Look at the constraint sprite's pixel at that position. If pixel == 0, the rule is "SAME": the grid cell at that neighbor offset must have the same color as required.
- If pixel != 0, the rule is "DIFFERENT": the grid cell must NOT have the same color as required.
- Check the actual grid cell at position (constraint.x + dx*4, constraint.y + dy*4) where (dx, dy) is the relative offset.
- **All constraints across all bsT sprites must be satisfied simultaneously** for the level to be won.

## Level Progression
**6 levels.** All grids are 32x32.
- Level 1 (THR): 3x3 grid of Hkx cells, 1 NTi cell at center, 2-color palette [9, 8]. Global pattern: single-cell (only center affected). StepCounter=32. Tutorial with flash indicator.
- Level 2 (hxv): 4x4 grid + extras, 1 NTi, 2 colors [9, 12]. 32 steps.
- Level 3 (Fmh): Larger grid with multiple bsT constraints, 2 colors [8, 12]. 96 steps.
- Level 4 (oea): 3-color palette [9, 8, 12]. 96 steps. 3 NTi cells.
- Level 5 (INW): 2 colors [14, 15]. 128 steps. Large grid with many constraints and NTi cells.
- Level 6 (DFx): Uses `ZkU` cells instead of `Hkx`. 2 colors [11, 14]. 128 steps.

Key data per level: `kCv` (step counter), `cwU` (color palette list), `elp` (global influence pattern as 3x3 grid of 0/1).

## No-Source Discovery Strategy
1. **First click:** Click a colored cell. Observe it (and possibly neighbors) change color.
2. **Observe:** Note which cells changed. Look at the global pattern -- if only center is 1, only the clicked cell changes. If NTi cells exist, clicking them may affect different patterns.
3. **Hypothesis:** Each click cycles colors on a pattern of cells. Small multi-colored 3x3 sprites are constraints showing what the final state should look like.
4. **Test:** Click the same cell again -- colors cycle through the palette (2 or 3 colors). Try clicking NTi cells -- different neighbor patterns.
5. **Strategy:** Read each constraint sprite's pattern: 0-colored positions around center = "same as center", other colors = "different from center". Solve the constraint satisfaction.

## Key Patterns
- Grid spacing: cells are 4px apart (3x3 cell + 1px gap, or placed at multiples of 4).
- Default influence pattern `elp = [[0,0,0],[0,1,0],[0,0,0]]` means only the clicked cell itself is affected.
- NTi cells override the pattern: their color-6 pixels define which neighbors are toggled.
- Color palettes: 2 colors = binary toggle, 3 colors = ternary cycle.
- Step counter shown as a bar at row 63 (bottom). Color 12 (pink) = remaining, 11 (teal) = spent.
- bsT constraint sprites have collidable=False when they represent "ghost" constraints (e.g., ASc, Bbi). Most are collidable=True.
- Background=4 (yellow), padding=4 (yellow).
- Constraint checking: pixel value 0 at position means "same color required", any other value means "different color required". This is checked against the center pixel's color for that constraint.
