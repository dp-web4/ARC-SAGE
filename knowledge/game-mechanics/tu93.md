## Interaction Model
- **UP/DOWN/LEFT/RIGHT** (ACTION1-4): Move the player one grid step (3 pixels) in that direction. Movement is validated against the maze -- the destination pixel in the maze sprite must be color 2 (walkable). Player's rotation is set to match movement direction (0=up, 90=right, 180=down, 270=left).
- **Three-phase step cycle** controlled by `iuubfszcoi`:
  - **Phase 0** (player input): Accept directional input. Move player 1 step. Each player direction is appended to all delayed entities' rotation queues. Set `iuubfszcoi = 1`. Step counter decrements.
  - **Phase 1** (player settling + arrow activation): Player (`albwnmiahg`) continues moving one pixel per frame until reaching a grid node (`sbkukvfrvl`: both row and column offsets divisible by `twsfmzbqkg`=6). At each non-node position, player moves 1 more step. When settled, overlapping enemies are shrunk/removed. Arrows (`vllvfeggte`) adjacent to the player activate (`ijipdchkji`: pixel[0,1] set to 11). Bouncers (`zzuxulcort`) take 1 step. Active delayed entities (`natiyqayts`) take 1 step. Transition to phase 2.
  - **Phase 2** (entity settling): All activated arrows, bouncers, and delayed entities continue moving until they reach grid nodes. Arrows and delayed entities that reach grid nodes destroy overlapping targets via `ojcejjkbwt`. Bouncers reverse direction at walls (`irydcdcqyd`). Delayed entities get newly activated if player is in range (`sdjbufarus`), and consume queued rotations. When all settled, check win/lose. Back to phase 0.
- **Step counter**: Decrements by 1 per directional input. Lose if it hits 0 or if no players remain.
- Each step call advances exactly ONE frame of the current phase.

## Win Condition
Checked at end of phase 2 (line 1264): ALL player sprites (`albwnmiahg`) must be at the SAME position as ANY exit sprite (`xboyuzuyxv`). Specifically:
```python
all([any([player.x == exit.x and player.y == exit.y for exit in exits]) for player in players])
```
Lose conditions: no player sprites remain on the board, OR step counter reaches 0.

## No-Source Discovery Strategy
1. **First action**: Press RIGHT. Observe if the player sprite moves rightward on the grid.
2. **Observe**: Note the maze pattern (color 2 = walkable paths). Look for the exit square (solid color 14). Look for other colored 3x3 sprites with directional markers.
3. **Hypothesis**: "Navigate the maze to reach the exit. Other sprites are entities that move/react."
4. **Test**: Move adjacent to an arrow entity (azure/color 8). It should activate and start moving. Watch it interact with targets.
5. **Test bouncer**: Move near a bouncer (magenta/color 12). Observe it moves and reverses at walls.
6. **Ready to solve**: Map the maze paths, identify entity positions and directions, plan a route to the exit.

## Key Patterns
- Grid unit = 3 pixels (`bsfndluqyd = 3`). Grid nodes at every 6 pixels (`twsfmzbqkg = 6`). Delayed entity activation range = 12 pixels (`gfjanriwfk = 12`).
- Grid node check (`sbkukvfrvl`): `i % 6 == 0 and x % 6 == 0` where i and x are offsets from maze origin.
- Center alignment check (`fqzuqmrwbm`): two sprites are "co-located" if their center points match: `(a.x + a.width/2 == b.x + b.width/2) and (a.y + a.height/2 == b.y + b.height/2)`.
- Maze walkability: `maze.pixels[dy + offset, dx + offset]` must equal 2. Out-of-bounds = blocked.
- Movement per step: exactly 1 pixel via `wkagrzgfwd` (adds +/-1 to x or y based on rotation). Entities take 3 frames to cross one grid cell and 6 frames to reach the next grid node.
- Arrow activation (`kqpvpzoaim`): each arrow (`vllvfeggte`) vs each player (`albwnmiahg`) via `lzfoxdhqxu` with distance 6. If arrow faces toward player at exactly 6px away, it activates (pixel[0,1] = 11), then moves 1 step.
- Target destruction (`ojcejjkbwt`): progressive shrinking -- height 3 -> replaced with 5x5 pattern, height 5 -> replaced with 7x7 pattern, height 7+ -> fully removed. Takes 3 entity arrivals at the same target to destroy it.
- Bouncer entity direction flip (`irydcdcqyd`): at end of phase 2, checks if next step in current direction would leave the maze or hit non-walkable cell. If so, reverses 180 degrees.
- Delayed entity activation (`sdjbufarus`): activates when ANY player is at distance 12 AND facing toward the entity. Stores `[rotation, rotation]` in queue. Each turn, pops from queue and applies rotation before moving.
- Delayed entity rotation queueing: every player input appends the chosen rotation to ALL delayed entities' queues, creating a delayed replay of the player's path.
- Entity direction: stored as sprite rotation (0=up/-y, 90=right/+x, 180=down/+y, 270=left/-x).
- Variable grid sizes per level: 39x38, 45x45, 45x45, 51x51, 49x49, 45x45, 45x45, 42x42, 51x51.
- Entity colors: Arrow = 8 (azure), Bouncer = 12 (magenta), Delayed = various, Exit = 14, Player = 9 (maroon).
