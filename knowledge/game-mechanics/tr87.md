# tr87 -- Pattern matching with symbol cycling on a two-row board

## Toolset
Available actions: [ACTION1 (UP), ACTION2 (DOWN), ACTION3 (LEFT), ACTION4 (RIGHT)].
Directional-only game -- no clicking. Implies: cursor-based navigation and symbol manipulation.

## Objects
| Tag/Name | Role | Description |
|-----|------|-------------|
| `gyrdjxybtcm` (A/B/C) | Background tile | 7x7 colored squares in 3 colors (A=10, B=7, C=11). Form the board grid. |
| `nxkictbbvzt` (A1-A7, B1-B7, C1-C7) | Symbol tile | 5x5 pattern tiles with 7 unique glyphs per color family (21 total symbols). These are the puzzle pieces. |
| `iqrduxrukrk` | Rule separator | 11x1 horizontal bars connecting left-side rule patterns to right-side rule patterns. |
| `nxkictbbvztedxeenecwqa` | Highlight overlay | 9x9 black square placed behind matched symbols during win animation. |
| `qvtymdcqear` (1/2/3) | Cursor indicator | Bracket sprites (top/bottom pair) showing which symbol is currently selected. Width varies: 5px (1 symbol), 12px (2), 19px (3). |
| `tjaqvwdgkxe` | Portal connectors | Small colored dots (1x1, colors 8/12/9/14) placed at symbol positions. Used for double/tree translation in later levels. |
| `background` | Divider | 2x1 scaled sprite separating the rule area (top) from the answer row (bottom). |

Color palette: background=2 (red), padding=3. Symbols always color 5 (grey). Background tiles: 10 (A/light orange), 7 (B/orange), 11 (C/cyan). Step counter on bottom row: green=1 remaining, yellow=4 used.

## Interaction Model
- **LEFT/RIGHT** (ACTION3/ACTION4): Move cursor to previous/next symbol in the answer row. In `alter_rules` mode, cycles through ALL symbol sets (both left and right sides of every rule). Cursor brackets update position.
- **UP/DOWN** (ACTION1/ACTION2): Cycle the currently selected symbol through its 7 variants. Symbol name format is `nxkictbbvzt{X}{N}` where X is color family (A/B/C) and N is 1-7. Cycling wraps: `(current_digit + direction - 1) % 7 + 1`. In `alter_rules` mode, cycles ALL symbols in the selected group simultaneously.
- **Step budget**: Each LEFT/RIGHT or UP/DOWN costs 1 step. Budget is 128 for levels 1-5, 256 for level 6. Lose when budget reaches 0.

## Win Condition
Method `bsqsshqpox()`: The top-row symbols define **rewrite rules** (left pattern -> right pattern). The bottom-left row is the **input sequence** and the bottom-right row is the **output sequence** (your answer). Win when: applying ALL rules sequentially to the input row produces exactly the output row.

Specifically:
1. Iterate through the input row (`zvojhrjxxm`) left to right.
2. For each position, try to match any rule's left-hand side against consecutive symbols starting at that position (match by sprite NAME).
3. If a match is found, the corresponding right-hand side must match the same span in the output row (`ztgmtnnufb`).
4. Continue until the entire input row is consumed.
5. If any position fails to match, return False.

In `double_translation` mode (L4): rules can chain -- if a rule's left side starts at a portal connector (`tjaqvwdgkxe`), follow the portal to find a second rule and compose them.

In `tree_translation` mode (L6): the right side of each rule must itself be the left side of another rule, and the final output uses those chained rules' right sides.

Win triggers a multi-frame highlight animation cycling through colors `[5, 8, 14, 15, 6, 9, 12, 0]`.

## Level Progression
6 levels. All 64x64 grid.
- **Level 1**: 3 rules, each with 1-symbol left and 1-symbol right. 2 color families (A, B). Simple 1:1 substitution.
- **Level 2**: 3 rules, 2 color families (B, C). More symbols in answer row (7+).
- **Level 3**: 3 rules, 2 color families (A, C). Larger answer rows (8+).
- **Level 4**: 4 rules with `double_translation=True`. Portal connectors chain two rules. 2-step rule application.
- **Level 5**: 2 rules with `alter_rules=True`. You can edit ALL symbol groups (rules + answer). Randomized initial orientations.
- **Level 6**: Complex -- `double_translation=True`, `alter_rules=True`, `tree_translation=True`. 3x3 rule grid with 3 color families. 256-step budget.

Key level data: `double_translation` (bool), `alter_rules` (bool), `tree_translation` (bool).

## No-Source Discovery Strategy
1. **First action**: Press LEFT or RIGHT. Observe the bracket cursor move between symbols in the bottom row.
2. **Observe**: Note the top section has paired symbol groups connected by horizontal bars (these are rules). The bottom has input and output rows.
3. **Hypothesis**: "This is a pattern/rewrite rule puzzle. I need to set the output row so that applying the top-row rules to the input produces the output."
4. **Test**: Press UP/DOWN to cycle a symbol. Note it changes to a different glyph. Try to match the rule patterns.
5. **Ready to solve**: Map out each rule (left pattern -> right pattern). Apply rules to the input row sequentially. Set the output row to match.

## Key Patterns
- Symbols cycle through 7 variants per color family. Each variant is a unique 5x5 glyph with different internal patterns of color-5 pixels and transparent (-1) pixels.
- Symbol names encode everything: `nxkictbbvzt{ColorFamily}{VariantNumber}`. Color families: A, B, C. Variants: 1-7.
- Symbol cycling formula: `new_name = old_name[:-1] + str((int(old_name[-1]) + direction - 1) % 7 + 1)`. Wraps 7->1 and 1->7.
- Rules are built from `iqrduxrukrk` separator bars. The bar's position defines the rule: symbols to the left of the bar form the left-hand side (LHS), symbols to the right form the right-hand side (RHS). Adjacent symbols are found by scanning horizontally at `iokndxodxw=7` pixel intervals.
- Rule matching (`iwbhnvdaao`): compares sprite NAMES starting at a position. Match is strict: `zdwrfusvmx[pos + i].name == gnxftjtboz[i].name` for all i.
- Rule application is greedy left-to-right: scan input row, try each rule at current position, consume matched symbols, advance, repeat.
- Layout: top half = rules (3x2 grids of colored background tiles with symbols). Bottom half = two rows: top row = input (answer row for editing), bottom row = fixed pattern.
- In `alter_rules` mode: the cursor can navigate ALL symbol groups (both rule LHS/RHS and answer rows). This means you edit the rules themselves, not just the answer. Cursor navigates a flat list of all sets.
- In `double_translation` mode: portal connectors (`tjaqvwdgkxe`, 1x1 colored dots) link rule chains. If a rule's LHS starts at a portal marked "1", follow the portal pair to find a "2" portal, locate the rule at that position, and compose: `LHS_combined = LHS1 + LHS2`, `RHS_combined = RHS2`.
- In `tree_translation` mode: each RHS element must itself be the LHS of another rule. The final output is composed of those rules' RHS elements. This creates a tree of rule applications.
- Random rotations are seeded per level (`random.Random(seed)`). Two seedings: first for initial rotation of all symbols, second for initial cycling of answer row symbols. Rotations CHANGE the visual orientation but matching uses NAME not visual -- so rotations are intentional visual obfuscation.
- Cursor bracket width adapts: `qvtymdcqear1` (5px, 1 symbol), `qvtymdcqear2` (12px, 2 symbols), `qvtymdcqear3` (19px, 3 symbols).
- Constants: `iokndxodxw = 7` (horizontal pixel spacing between symbols), `kjgicbtgrt = 7` (cycle length), `fiwynmpoeb = 1` (progress bar fill color), `efblfzysop = 4` (progress bar empty color).
- Win animation: cycles matched symbols through 8 colors `[5, 8, 14, 15, 6, 9, 12, 0]` frame by frame, with black highlight squares (`nxkictbbvztedxeenecwqa`, 9x9) placed behind each matched rule's symbols.
- Background tile colors: A=10, B=7, C=11. Answer row tiles match the expected color family.
- Step budget is generous (128-256) but every cursor move and every symbol cycle costs 1 step.
