# s5i5 -- Slider-card rotation puzzle with path routing

## Toolset
Available actions: [ACTION6 (CLICK only)]. Click-only game -- no directional movement.
Implies: point-and-click puzzle where all interaction is through clicking specific objects.

## Objects
| Tag | Role | Description |
|-----|------|-------------|
| `gdgcpukdrl` | Interactive (slider) | Colored bar sprites (7x7 blocks, colors 10/7/11 = A/B/C). Clicking left/right of center grows/shrinks associated cards. |
| `agujdcrunq` | Interactive (card/arm) | Colored rectangular cards that extend from sliders. Width 3 (`bjntvocxdv`), variable length. Have orientation (0/90/180/270) determined by which edge has color 3 pixels. |
| `myzmclysbl` | Interactive (rotate button) | Small colored buttons. Clicking triggers rotation of all same-colored cards by 90 degrees. |
| `zylvdxoiuq` | Static (connector) | Connectors that link cards to sliders via collision detection. |
| `cpdhnkdobh` | Target | Goal markers (sprite `nwtrqgdsmb`, 3x3 color-5 block). Win when all targets overlap with `zylvdxoiuq` connectors. |

Color palette: background=5 (grey), padding=3. Cards use various colors. Step counter bar on row 63 (green=3 remaining, yellow=4 used).

Custom Camera: `jcevyecaju` -- renders at 64x64 directly (no scaling), standard sprite blitting with transparency (-1).

## Interaction Model
- **CLICK**: Converted via `display_to_grid`. Two outcomes:
  1. **Click on rotate button** (`myzmclysbl`): reads the center pixel color, finds all cards (`agujdcrunq`) of that color, saves their state, then rotates them 90 degrees (with special double-rotation logic if card orientation differs by 180 from parent). If rotation causes collision between cards, the move is reverted on next step.
  2. **Click on slider** (`gdgcpukdrl`): determines if click is left or right of slider center. Left = shrink card by 1 unit, right = grow by 1 unit. Minimum size is 1 unit. Card resizing respects orientation (grows in the direction the card points). If resize causes collision, reverted.
- **Undo system**: `acgkuydgqx` dict stores cloned sprite states. If a move causes collisions (`ulzimrggno`), the next step restores all saved positions/pixels.
- **Step counter**: Decrements by 1 per click. Lose when counter hits 0.

## Win Condition
Method `vodebmynqs()`: For EVERY sprite tagged `cpdhnkdobh` (targets), there must exist at least one sprite tagged `zylvdxoiuq` (connector) at the SAME (x, y) position. All targets must be covered. Checked after every action.

## Level Progression
8 levels. All use 64x64 grid.
- **Level 1**: Simple -- 2 targets, no Children hierarchy, 50 steps.
- **Levels 2-8**: Introduce `Children` data -- parent-child card relationships where moving/resizing parent also moves children recursively (`qdqfmefxmo`). Step counters: 50-200.
- **Levels 5-8**: Add NTi button sprites (`nqfjqnibax`, `qukdkhalyj`, `snpntdhtic`, `xxctmacvll`) positioned in the bottom row area, likely additional rotate buttons.
- **Level 8**: Most complex -- 8 parent-child pairs, 200 steps.

Key level data: `StepCounter` (int), `Children` (list of [parent_name, child_name] pairs).

## No-Source Discovery Strategy
1. **First click**: Click on a colored bar (slider area). Observe if a card grows or shrinks.
2. **Observe**: Note the colored arm extending from the slider. Check if the step counter at the bottom decreases.
3. **Hypothesis**: "Clicking left/right of slider center shrinks/grows attached cards. Colored buttons rotate same-colored cards."
4. **Test**: Click a colored button (small square). Observe if cards of that color rotate 90 degrees.
5. **Ready to solve**: Once you see targets (small squares), note their positions and figure out how to route cards to cover all targets by resizing and rotating.

## Key Patterns
- Grid: always 64x64, rendered directly at native resolution.
- Card width is always 3 pixels (`bjntvocxdv = 3`). Length is multiples of 3 (height or width depending on orientation).
- Card orientation detection (`fhkoulsvoi`): determined by which edge of the card has color-3 (padding) pixels:
  - Bottom edge = 0 degrees (card points up)
  - Left edge = 90 degrees (card points right)
  - Top edge = 180 degrees (card points down)
  - Right edge = 270 degrees (card points left)
- Rotation pivot (`ayrfrgnhfh`): cards rotate 90 degrees CCW using `np.rot90`. The pivot point depends on the card's current orientation -- it rotates around the anchor edge corner.
- Card resizing (`ldmrjbrvwa`): replaces entire pixel array. New size = `index * bjntvocxdv`. The anchor edge (color-3 row/column) is preserved. Children are moved by the displacement delta.
- Children move recursively -- `qdqfmefxmo` propagates (dx, dy) to all children in `enplxxgoja` dict.
- Parent-child linkage (`enplxxgoja`): built during `on_set_level` from `Children` data. Also includes collision-based linking between cards and connectors.
- Collision detection between cards (`ulzimrggno`) causes automatic revert -- invalid moves are "free" (but still cost a step). The revert happens on the NEXT step call.
- Rotate button double-rotation: if the card's orientation minus 90 differs from its parent's orientation by 180 degrees, the card is rotated TWICE (line 2209-2211). This handles cards pointing "backward" relative to their parent.
- Coordinate mapping: `display_to_grid` is the custom Camera's method. Since the camera renders at 64x64 with no scaling, grid coords approximately equal display coords.
- Colors: slider A=10, B=7, C=11. Cards inherit color from their slider's pixel values.
- Step counter constants: `bjntvocxdv=3` (card width), `smquggholb=5` (slider reading stride), `rqyqtrpuud=3` (anchor color), `hbwmfaubdy=1`, `zahyxpjwoj=4` (slider end markers), `cczmmxbepc=2`.
- The `thvbtaauus` function reads slider position by scanning pixels at stride-5 intervals to find where color changes to 1 or 4, returning the card index.
