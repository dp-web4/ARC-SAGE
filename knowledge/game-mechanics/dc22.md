# dc22 -- Grid navigation puzzle with crane, bridges, pressure plates, and color cycling

## Toolset
Available actions: [1, 2, 3, 4, 6] (UP, DOWN, LEFT, RIGHT, CLICK).
Hybrid game: the avatar moves with directional keys, and a separate crane/cursor mechanism is operated by clicking on UI buttons rendered as sprites. No ACTION5.

## Interaction Model
**Avatar movement (ACTION1-4):** Moves avatar by `qqsswktiui`=2 pixels in the chosen direction. Standard collision detection (collidable sprites block movement). After moving:
- Checks if avatar is standing on a valid surface (`uxwpppoljm` -- finds any intangible sprite under the avatar, excluding path/nxhz). If not supported, **fall animation** triggers.
- Checks for `zbhi` (key) overlap: if found, permanently unlocks matching letter-tag `jpug` gates and removes the key.
- Checks win condition.
- Updates pressure plate state (`mbszrqqnqm`).
- Decrements step counter by 1. Lose if counter hits 0.

**Fall animation (`uuehztercxf`):** Avatar visually dissolves over 14 frames (pixels replaced with color 4 progressively). At frame 14, game restores the last checkpoint. Costs 20 step-counter units (via `yepbymuune()`). If steps run out during fall, game is lost. The render_interface shows a shrinking circle effect during the fall.

**Click interaction (ACTION6):** Converts display coordinates to game coordinates via `camera.display_to_grid`. Saves checkpoint before processing. Two main click targets:

1. **`jpug` gate-buttons:** Clicking activates color group logic:
   - If tagged `itki-color-jpug`: cycles all `itki-color-cycle` tiles through the 4 color variants (itkiupry -> itkibgeg -> itkizfrq -> itkijbyz -> wrap).
   - Otherwise: finds all sprites sharing the same letter-tag. For each, swaps to the next numbered variant (e.g., variant 1 <-> variant 2). Variants have different interaction modes: `gigzqgcfncq`=intangible, `invisible`=invisible, `jpug`=intangible, else tangible.
   - After cycling, if avatar is no longer supported, fall triggers.

2. **`sys_click` buttons (crane controls):**
   - Direction buttons (`zmjbupyjfyb`/`jffakyxiury`/`vbdduyutyiw`/`ghqmfnmmgrz`): Move crane by 4 pixels in the indicated direction. For `uzungcfexgf` mode (hhxvnxhz variant), validates against path grid. For standard mode, validates against fixed bounds (x: 0-3 at y=0 or y=3; y: 0-3 at x=0). If valid, crane moves and any attached bridge/object moves with it. If invalid, **bounce animation** plays (crane moves out then back, 2 frames).
   - `bynyvtuepbt` (grab): If crane is empty, attempts to grab: first checks for an hhxv bridge at crane position (uzungcfexgf mode), then checks for a `bynyvtuepbt-object`. If found, attaches it. If nothing found, **flash animation** plays (5 frames of alternating crane sprites). If crane is already carrying, this is effectively a no-op.
   - Bridge collision: if moving the attached bridge would overlap another bridge, move is rejected and bounce animation plays.
   - After each crane operation, pressure plates are re-evaluated.

**Checkpoint system (`xcrifaqaiy`):** Before each click action, saves a full snapshot of all sprite states. If the avatar falls, restores this snapshot (`ycfbtkckze`), undoing the last click-action's effects (but costing 20 steps).

## Win Condition
Method `uhqbwfdkff()` (line 2933):
```python
self.fdvakicpimr.x == self.bqxa.x and self.fdvakicpimr.y == self.bqxa.y
```

Player position must exactly match goal position. Checked after every directional move. On win, calls `self.next_level()`.

Loss: step counter depleted. Falls cost 20 steps; normal moves and clicks cost 1 step each.

## Level Progression
6 levels. Grid sizes from 64x44 to 64x64. Step budgets: 128, 192, 192, 192, 256, 1024.

- **Level 1**: Simple. Avatar, exit, floor blocks, gate buttons. No crane. Walk to goal. 128 steps.
- **Level 2**: Introduces crane (`nxhz`), bridge pieces (`hhxv`), and arrow-button UI. Must reposition a bridge to create a walkable path. 192 steps. Also introduces invisible-gate jpug variants.
- **Level 3**: Adds `itki` color-cycling tiles, `zbhi` keys, more complex gate logic. Multiple floor regions connected by bridges and gates. 192 steps.
- **Level 4**: Adds pressure plates. Standing on them toggles gates. Requires careful sequencing. 192 steps.
- **Level 5**: Combines all mechanics. Multiple bridge pieces, color cycles, keys. 256 steps.
- **Level 6**: Full complexity. Uses `hhxvnxhz` crane variant (width/height-based anchor offsets), `bynyvtuepbt` grab objects, color-cycling itki tiles, pressure plates, path-grid-validated crane movement. Special initialization adds `itki-color-cycle` tag to specific sprite. Very large budget (1024 steps).

## No-Source Discovery Strategy
1. Move with arrow keys. Avatar moves 2px per step. Note floor tiles and walls (1-2 actions).
2. Walk off a floor tile intentionally. Observe fall animation and checkpoint restore. Note the step cost -- watch the progress bar jump (1 action).
3. Click on the arrow-button UI panel (right side of screen). See the crane/cursor move on the board (1-2 actions).
4. Click the grab button. If the crane is over a bridge piece, observe it attaches. Move the crane -- bridge follows (2 actions).
5. Click colored gate buttons if present. Observe tile visibility changes (1-2 actions).
6. Walk avatar onto pressure plates. Observe gates toggling (1 action).
7. Navigate avatar to goal (color 11, 2x2). Confirm level completion (1 action).

## Key Patterns
- **Crane and avatar are independent systems.** You move the avatar with directions and the crane with clicks. Plan crane operations first, then walk.
- **Falls are expensive (20 steps) but recoverable.** The game restores the last click-checkpoint, so you can experiment with crane positions.
- **Bridge pieces are the core puzzle.** Crane-move bridges to span gaps between floor tiles so the avatar can walk across.
- **Color-cycling tiles gate progression.** Cycle through 4 variants: itkiupry, itkibgeg, itkizfrq, itkijbyz. Each variant may change platform tangibility.
- **Pressure plates are position-dependent.** Avatar must remain on the plate for gates to stay open. Plan routing carefully.
- **Keys (zbhi) are permanent.** Once collected, their gates stay open forever. Prioritize key collection.
- **Crane path grid constrains crane movement.** `bg`/`path` tiles define where the crane can go. Not all positions are reachable.
- **Bridge collision prevents stacking.** Order of bridge moves matters when multiple bridges exist.
- **Level 6 uses hhxvnxhz variant** where the crane is a bridge-like object with width/height-based anchor offsets, making spatial planning more complex.
- **Step budget varies dramatically.** Level 1 has 128 steps; level 6 has 1024. Later levels need more exploration but also have more budget.
