## Interaction Model
- **CLICK**: Converted via `display_to_grid(x, y)`. Three click targets:
  1. **Arrow bits** (`baxznkbwix`): Toggle individual bits on/off. Each column encodes an instruction number as a binary sum (bit 0 = 1, bit 1 = 2, etc.). The instruction set (command map at line ~2170):
     - 0: flash/highlight block
     - 1: move left, 2: move right, 3: move down, 33: move up
     - 5: rotate +90, 6: rotate -90, 7: rotate 180, 16: rotate 270
     - 8: scale up, 9: scale down
     - 10-13: double-distance moves
     - 14/15/63: color changes
  2. **Tab selector** (`qgqvgxjtau`): Switch active program tab (levels with multiple programs).
  3. **Confirm button** (`rlqfpkqktk`): Execute both programs (left panel runs its program, right panel checks for win).
- **Animation system** (`nwjrtjcxpo`): When programs execute, the `zhxlrqdrcn` method steps through each instruction one frame at a time. Each column's bits are decoded, the corresponding command runs, and the block animates. Animations block input until complete.
- **Progress bar** (`yrddcregck`): Moves left by 1 pixel per action (or 0.5 pixel for level >= 5). Lose when the bar runs out (`fisapprjnh` returns False).

## Win Condition
Method `woaawujmbs()` -> `yxabhsirzl` property on the RIGHT panel (`xsseeglmfh`): The movable block must match the target in ALL of: x position, y position, scale, rotation, AND color (`dtxpbtpcbh`). Both panels must satisfy this condition -- the right panel is the one checked for the main win condition.

After winning animation completes, `rarvldaizc` flag triggers `next_level()`.

## Level Progression
~7+ levels. 64x64 grid throughout.
- **Level 1**: Simple -- 5 arrow columns, no program data, no tab selectors. Just the basic movement instruction encoding.
- **Level 2+**: Introduces `Programs` (initial instruction values), `Positions` (block starting offsets), `Rotations`, `Scales`, `Reset` flags. Multiple program tabs via tab selectors.
- **Level 2**: 2 programs, each with 4 instructions. Introduces portals and boundary walls.
- **Level 3+**: More columns (6+), more programs, arrow directions for movement, scale changes, and color changes.
- **Level 5+**: Progress bar slows to half speed.

Key level data: `Programs` (list of int lists -- initial instruction values per tab), `Positions` (list of [dx, dy] offsets), `Rotations` (list of ints), `Scales` (list of ints), `Reset` (list of bools).

## No-Source Discovery Strategy
1. **First click**: Click on the arrow-shaped toggles in the lower-left panel. They should flip between two states (on/off).
2. **Observe**: The arrows form columns. Each column represents one instruction step. Note there are two panels (left = programming, right = result display).
3. **Hypothesis**: "Binary encoding of arrows in each column determines movement instructions. Confirm button executes."
4. **Test**: Set a simple pattern (single arrow in first column) and click the confirm button. Watch the block move.
5. **Ready to solve**: Once you see the block animate and can map arrow patterns to movements, program the sequence to move the block to the target position.

**CRITICAL BUG NOTE**: Level 1 has arrow selectors present BUT they start with default values. The tablet tagged `ilkhmcqzxr` (readonly) may block clicks. If clicks seem unresponsive, look for the writable tablet (without `ilkhmcqzxr` tag).

## Key Patterns
- Instruction encoding: each column (`yfozlxgytl`) contains multiple arrow bits (`mzydzdjehq`). The column value `kbswvermjk` is computed as `sum(1 << i for i, m in enumerate(bits) if m.hokejgzome)`. This is a bit-field: bit 0 = +1, bit 1 = +2, bit 2 = +4, etc.
- Full command map (from `dfguzecnsr` dict):
  - 0: highlight/flash block
  - 1: move left (-dx, 0), 2: move right (+dx, 0)
  - 3: move down (0, +dy), 33: move up (0, -dy)
  - 5: rotate +90, 6: rotate -90, 7: rotate 180, 16: rotate 270
  - 8: scale +1, 9: scale -1
  - 10, 11: move right double distance, 12, 13: move left double distance
  - 14: color change to `cegfmryrnx`, 15: color change to `qeylsfmmom`, 63: color change to `bynlrmesca`
- Portals toggle visibility every 3rd instruction step (`pdawcsxhlx` at `txgixiygnw % 3 == 2`). When a portal becomes visible and collides with the block, the block is "consumed" (`oqquecbwaw` -- hidden, replaced with ghost sprite).
- Movement distance: `oocxrjijjq` (grid unit size, defined outside the class, likely 5 based on sprite spacing of 5px between grid columns).
- Block collision with walls (`nyxtqtuuny`) reverts movement (move, then check, then undo). Collision with portals (`cbbgkmkxku`) removes the block.
- Scale collision: scaling up that causes wall collision is reverted. Scaling that causes portal collision removes block.
- Hole-in-wall sprites (`wpilazztrp`): hidden when block occupies same position at same scale. This is visual feedback showing the block "fits" in the wall gap.
- Progress bar (`yrddcregck`): moves left by 1 pixel per action (0.5 pixel for level_index >= 5). Lose when bar's right edge passes the background bar's right edge.
- Two-panel architecture: `kmhfvaisxu` (left/first panel) and `xsseeglmfh` (right/second panel). Both are `dzuwewgzxj` instances. Win is checked on the RIGHT panel only.
- Tab switching (`dnkhuycwgt`): saves current arrow states to `lhhjnfuvau`, loads new tab's program data, resets block position/rotation/scale to tab's stored values.
- Arrow bit colors: on = `uaqblvbfgj`, off (read-only) = `jsumigcuvq`, off (writable) = `dhvnsgsstf`. Highlight = `vxmzftbjat`.
- Animation execution: `xfmltsqjmt` triggers both panels. Each panel reads its column values into `nensmzujat`, then `zhxlrqdrcn` steps through one instruction per frame, executing the corresponding lambda from `dfguzecnsr`.
- The readonly tablet (tagged `ilkhmcqzxr`) ignores clicks (`csaytdwcpe` returns False if `vguslsxvft` is True).
